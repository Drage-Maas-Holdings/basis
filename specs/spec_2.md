SCHEMA DEFINITIONS:

Table: users
- id          TEXT PRIMARY KEY (cuid, generated by the application layer)
- email       TEXT NOT NULL UNIQUE
- name        TEXT NOT NULL
- password    TEXT NOT NULL (hashed, never plaintext)
- created_at  INTEGER NOT NULL (Unix timestamp in milliseconds)
- updated_at  INTEGER NOT NULL (Unix timestamp in milliseconds)

Table: contacts
- id          TEXT PRIMARY KEY (cuid)
- name        TEXT NOT NULL
- email       TEXT
- phone       TEXT
- company     TEXT
- notes       TEXT
- owner_id    TEXT NOT NULL REFERENCES users(id)
- created_at  INTEGER NOT NULL
- updated_at  INTEGER NOT NULL

Table: deals
- id          TEXT PRIMARY KEY (cuid)
- title       TEXT NOT NULL
- stage       TEXT NOT NULL CHECK (stage IN ('new', 'qualified', 'won', 'lost'))
- value       INTEGER (deal value in minor currency units, e.g. cents; nullable)
- contact_id  TEXT REFERENCES contacts(id)
- owner_id    TEXT NOT NULL REFERENCES users(id)
- created_at  INTEGER NOT NULL
- updated_at  INTEGER NOT NULL

Table: tasks
- id          TEXT PRIMARY KEY (cuid)
- title       TEXT NOT NULL
- due_at      INTEGER (Unix timestamp in milliseconds; nullable)
- completed   INTEGER NOT NULL DEFAULT 0 (0 = false, 1 = true)
- contact_id  TEXT REFERENCES contacts(id)
- deal_id     TEXT REFERENCES deals(id)
- owner_id    TEXT NOT NULL REFERENCES users(id)
- created_at  INTEGER NOT NULL
- updated_at  INTEGER NOT NULL

Table: interaction_logs
- id          TEXT PRIMARY KEY (cuid)
- type        TEXT NOT NULL CHECK (type IN ('email', 'call', 'meeting', 'note'))
- body        TEXT NOT NULL
- contact_id  TEXT REFERENCES contacts(id)
- deal_id     TEXT REFERENCES deals(id)
- logged_by   TEXT NOT NULL REFERENCES users(id)
- created_at  INTEGER NOT NULL (append-only; no updated_at)

ACCEPTANCE CRITERIA:
1. The placeholder table is removed from api/src/db/schema.ts
2. All five tables are defined in api/src/db/schema.ts using Drizzle's
   sqliteTable
3. All foreign key relationships are defined in the schema
4. npm run db:generate (from within api/) produces a new migration file in
   api/src/db/migrations/
5. npm run db:migrate (from within api/) applies the migration without error
6. All five tables exist in the SQLite file with the correct columns and
   constraints, verifiable via npm run db:studio or sqlite3 inspection
7. npm run build (tsc, from within api/) completes with zero errors after
   schema changes

CONSTRAINTS:
- Use Drizzle ORM syntax (sqliteTable, text, integer) — do not write raw SQL
- Do not use Drizzle's $defaultFn for cuid generation in the schema; cuid values
  are generated in the application layer before insert
- INTEGER columns are used for all timestamps (Unix ms) and boolean fields
- The interaction_logs table must not have an updated_at column

DESIGN DECISIONS:
- Timestamps are INTEGER (Unix ms) not TEXT; this keeps range queries and
  sorting free of casting
- deals.value is stored in minor currency units (e.g. cents) as INTEGER;
  SQLite REAL introduces floating point precision issues for money values
- cuid generation happens in the application layer before insert; using
  Drizzle's $defaultFn hides the value from TypeScript's type inference on
  insert, which causes friction when auth needs to create a user and return
  a typed response
- interaction_logs has no updated_at column; append-only is enforced at the
  schema level by omission, not convention

OUT OF SCOPE:
- Seed data
- Any API routes or controllers
- Authentication or session handling (covered in Spec 03)
- Index creation beyond what Drizzle generates for primary keys and unique constraints

EDGE CASES:
- A task must be linkable to a contact, a deal, both, or neither; both foreign
  keys are nullable and no constraint enforces that at least one is set
- An interaction log must be linkable to a contact, a deal, or both; same rule
  applies
- The deals.value column is nullable to allow deal creation before a value is known
